# RDK__X5 Session Handoff

## 1. 项目目标

这个工程当前的目标是：

- 在 ESP32-S3 + 4.3 寸触摸屏上显示一个机器人脸部界面。
- 使用 LVGL 绘制表情。
- 使用 GT911 触摸控制器响应点击/长按。
- 通过 `robot_link -> emotion_engine -> display_service` 这条链路驱动最终表情。
- 参考 `emoji/5.html` 中的 8 个表情设计，完成触摸交互和自动行为。

当前已经实现的核心方向是：先让触摸和基础显示稳定，再把图片参考中的 8 个表情行为映射到显示服务中。

## 3. 系统运行链路

系统主循环在 `main/app_main.c` 中，启动顺序如下：

1. `bsp_board_init()`
2. `display_service_init()`
3. `emotion_engine_init()`
4. `robot_link_init()`
5. 主循环中每 50ms 执行一次：
   - `robot_link_poll()`
   - `emotion_engine_request_state()`
   - `emotion_engine_update()`
   - `display_service_render()`

对应文件：

- `main/app_main.c`
- `components/robot_link/robot_link.c`
- `components/emotion/emotion_engine.c`
- `components/display/display_service.c`

## 4. 本轮对话前已完成的修复

### 4.1 GT911 触摸不响应

问题现象：

- 屏幕触摸没有反应。
- 初始化日志出现 `ESP_ERR_INVALID_ARG`。

定位结果：

- GT911 的 I2C panel IO 配置里使用了不兼容的 `scl_speed_hz` 赋值，导致初始化失败。

修复方式：

- 在 `components/display/display_service.c` 的 `display_service_touch_init()` 中，保留 `ESP_LCD_TOUCH_IO_I2C_GT911_CONFIG()` 的默认结构，不再手动覆盖 `scl_speed_hz`。

结果：

- 触摸初始化路径恢复正常。

### 4.2 表情偶发变红、抖动

问题现象：

- 脸部会周期性变成红色错误脸。
- 用户观察到“有时候表情变红，会抖动”。

定位结果：

- `robot_link_poll()` 之前返回了轮换状态，其中周期性会触发 `ROBOT_STATE_ERROR`。

修复方式：

- 将 `components/robot_link/robot_link.c` 中的 `robot_link_poll()` 简化为稳定返回 `ROBOT_STATE_IDLE`。

结果：

- 取消了周期性错误状态注入。
- 显示不再无故进入红脸错误态。

## 5. 本轮新增实现

本轮核心改动集中在：

- `components/display/display_service.c`

目标是把 `emoji/5.html` 中的 8 个表情逻辑映射到固件里。

### 5.1 已接入的表情状态

当前显示层已加入这些状态：

- `DISPLAY_FACE_HAPPY`
- `DISPLAY_FACE_SHY`
- `DISPLAY_FACE_EXCITED`
- `DISPLAY_FACE_LOOK_LEFT`
- `DISPLAY_FACE_LOOK_RIGHT`
- `DISPLAY_FACE_BLINK`
- `DISPLAY_FACE_SLEEP`
- `DISPLAY_FACE_ERROR`

另外新增了内部辅助状态：

- `DISPLAY_FACE_NONE`

用于表示“当前没有临时触发表情”。

### 5.2 触摸行为

已实现的触摸行为：

- 点击上方区域：
  - 进入 `shy`
  - 数秒后自动回到 `happy`
- 点击中间区域：
  - 立即恢复 `happy`
- 长按 `1.5s`：
  - 进入 `excited`
  - 短暂停留后自动回到 `happy`
- 点击下方区域：
  - 不切换成指定表情
  - 但会记录一次交互，用于阻止立刻休眠或立刻触发自动行为

触摸分区逻辑为纵向三分区：

- `top`
- `middle`
- `bottom`

实现点：

- `display_service_touch_band_from_point()`
- `display_service_update_touch_locked()`

### 5.3 自动行为

已实现的自动行为：

- `look left`
  - 空闲时随机触发
  - 触发间隔 `3~6 秒`
- `look right`
  - 空闲时随机触发
  - 触发间隔 `3~6 秒`
- `blink`
  - 空闲时随机触发
  - 触发间隔 `10~20 秒`
- `sleep`
  - `3 分钟`无交互进入休眠

自动行为约束：

- 自动行为不会覆盖触摸产生的临时表情。
- 自动行为在触摸开始时会被清空并重新排期。
- 休眠优先级高于普通默认表情，但错误态仍优先于休眠。

实现点：

- `display_service_random_range_us()`
- `display_service_schedule_gaze_locked()`
- `display_service_schedule_blink_locked()`
- `display_service_visual_face_from_state()`

### 5.4 表情绘制实现方式

本轮没有重做整套 UI，而是在已有 LVGL 对象基础上扩展：

- 左眼、右眼
- 左右高光
- 左右腮红
- 嘴线对象
- 睡眠标签

新增对象：

- `s_left_shade`
- `s_right_shade`
  - 用于实现 `look left / look right` 时的遮罩感
- `s_mouth_dot`
  - 用于实现 `excited` 的椭圆嘴型

实现策略：

- 通过切换对象尺寸、位置、透明度、阴影和隐藏状态，模拟 `emoji/5.html` 中的 SVG 效果。
- 不是逐像素还原 SVG，但行为和视觉意图已经对齐。

## 6. `emoji/5.html` 的作用

`emoji/5.html` 是本轮最关键的视觉参考文件。

它定义了：

- 8 个目标表情
- 触摸与自动行为分组

当前固件实现已经把这份 HTML 的主要交互和表情语义映射到 C 代码中。

## 7. 当前各模块状态

### 7.1 `robot_link`

当前状态：

- 仍是占位 stub。
- `robot_link_poll()` 现在固定返回 `ROBOT_STATE_IDLE`。

意义：

- 保证显示层稳定，不被上层随机错误状态扰动。

后续如果接入 UART、Wi-Fi、MQTT 或语音链路，可以再把真实状态输送回去。

### 7.2 `emotion_engine`

当前状态：

- 仍是轻量状态机。
- 上电先进入 `BOOTING`。
- 大约 `1.5s` 后自动切到 `IDLE`。
- 如果收到新请求状态，就切到请求值。

意义：

- 提供一个稳定的过渡层，让 `display_service` 不直接依赖 `robot_link` 的原始输入。

### 7.3 `display_service`

当前状态：

- 是本轮开发重点。
- 已承担：
  - LCD 初始化
  - GT911 初始化
  - LVGL 初始化
  - 表情对象创建
  - 触摸读取
  - 表情决策
  - 自动行为调度
  - 最终界面渲染

## 8. 当前实现中的优先级关系

显示层当前采用的是“基础状态 + 触摸临时状态 + 自动行为”的组合思路。

优先级大致如下：

1. `ROBOT_STATE_ERROR`
2. 长时间无操作触发的 `sleep`
3. 触摸临时表情
4. 自动行为
5. 默认 `happy`

这样做的目的：

- 保证错误态不会被普通表情覆盖。
- 保证手势表情不会被自动行为抢走。
- 保证休眠能够在真正长时间无交互时生效。

问题现象：

结果：

- `task_wdt` 崩溃不再复现

问题现象：

- 启动日志里会出现 `Unable to initialize the I2C address`

修复方式：

- 在 GT911 驱动里先检测当前地址是否已经可用
- 如果已经可用，则跳过内部地址切换
- 将 warning 改成信息日志，明确说明使用的是预选地址

结果：

- 启动日志变为正常信息提示
- 更符合实际硬件行为

## 17. 本轮补充进展（1.0.1 -> 1.0.2）

这一轮继续围绕“左右看表现不自然”做了针对性调整，目标是把 `v1.0.1` 中偏特效化的 gaze 表现改成更直接的整体脸部偏移方案。

### 17.1 左右看视觉方案重构

问题现象：

- `v1.0.1` 中用于 `look left / look right` 的伪发光描边、叠加遮罩和白色高光点观感不够自然
- 视线表达更像“眼睛上叠了一层效果”，而不是整张脸真的在看向左侧或右侧

调整方式：

- 在 `components/display/display_service.c` 中移除了左右看依赖的白色高光点对象
- 移除了为左右看准备的左右遮罩对象，不再用叠层覆盖来制造方向感
- 将左右看改为表情脸整体横向偏移：
  - 左右眼一起偏移
  - 嘴型一起偏移
  - 腮红一起偏移

## 18. 本轮补充进展（启动开场动画 / momo）

这一轮继续在现有 `display_service` 基础上增加了一个一次性的开机开场动画角色 `momo`，目标是在不破坏现有触摸与自动行为的前提下，让启动阶段更有“角色感”。

### 18.1 启动阶段新增 `momo` 像素形象

- 在 `components/display/display_service.c` 中新增了一个粉色像素风 `momo` 覆盖层
- `momo` 不是替换原本脸部 UI，而是作为独立前景对象叠加在主界面上
- 角色主体采用轻量矩形块拼接，而不是额外图片解码，尽量避免引入较重资源路径

### 18.2 开机时序调整

当前启动阶段的实际显示逻辑变为：

1. 上电后，底层脸部先保持 `sleep`
2. 约 `2 秒`后，`momo` 从屏幕上边界开始出现
3. 先只露出一点，再到半身，再到全身，整个过程为平滑位移
4. 全身露出后继续向下平滑落到底部，并从下边界消失
5. `momo` 消失后，系统回到原有默认 `happy` / `idle` 逻辑
6. 后续触摸交互、自动左右看、眨眼、休眠等行为保持原样

### 18.3 `momo` 下落时的表情变化

为了让出场过程更有生命感，在下降阶段补充了轻量表情切换：

- `happy`
- `blink`
- `surprised`
- `shy`

这些变化只作用于 `momo` 本身，不影响原本主脸状态机。

### 18.4 启动期输入策略

为避免开场动画和原交互抢状态，启动序列进行中增加了保护：

- 启动动画期间暂时屏蔽触摸输入
- 不让触摸提前打断开场序列
- 动画结束后恢复原有触摸逻辑

### 18.5 实现方式说明

当前实现仍然遵循“尽量轻量”的方向：

- 没有引入额外图片资源解码链路
- 没有改动 `emotion_engine` 的主体状态机
- 主要在 `display_service` 内部增加：
  - 启动时序判断
  - `momo` 像素对象创建
  - `momo` 表情切换
  - 启动期间的触摸屏蔽

这样可以把改动范围控制在显示层，降低对现有交互链路的影响。

### 18.6 本轮后续微调记录

在首次加入 `momo` 后，又根据实机反馈继续做了两类修正：

- 将 `momo` 从较大的初版缩小为更轻量的小像素体型，避免遮挡主体画面
- 重画了像素轮廓与五官布局，使其更接近参考图的粉色小形象风格
- 将启动动画缓动改为更平滑的 S 曲线
- 将主循环刷新间隔从 `50ms` 调整到 `20ms`，减轻开场位移的跳帧感
- 对 `momo` 的位置更新和表情更新增加状态缓存，只有在确实发生变化时才刷新对象，减少重绘抖动
- 后续又继续将 `momo` 再缩小一档，并把启动期间的底层 `sleep` 脸改为只在进入时绘制一次，不再每帧重复刷新，从而进一步压低开场动画的重绘负担
- 继续将 `momo` 的上方出场改成单段连续缓动，去掉分段 reveal 在边界处的顿挫感；同时给主脸眼睛、腮红、嘴、口型点和 `touch` 状态文案增加变化缓存，减少后续各种表情切换时的无效重绘
- 将主循环刷新间隔进一步从 `20ms` 调整到 `16ms`，在减少重绘后尽量靠近 60fps 的更新节奏
- 根据后续实机反馈，已移除底部状态/提示/触摸三行文字，并保留触摸逻辑本身，以进一步减少底部区域的无效文字重绘
- 在不砍任何现有动效的前提下，继续让主脸在“表情目标已到位且当前未处于眨眼变化中”时直接跳过本帧 `apply`，避免稳定状态下反复走整套对象更新
- 将 GT911 触摸采样从每帧读取改为独立的轻量轮询缓存，渲染仍保持 `16ms` 节奏，但触摸硬件读取不再强绑定每一帧，减少 I2C 侧对动画流畅度的拖拽
- 继续将 `display_service_render()` 改成“按需进入 LVGL”模式：当开场动画、眨眼、触摸按压、长按判定、表情过渡或定时事件未到期时，直接跳过本轮 UI 更新；只有真正需要刷新时才重新加锁并更新对象，从而进一步降低静止画面下的 CPU 占用和重绘压力
